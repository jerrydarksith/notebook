# Архітектура проєкту

## Призначення

Це персональна система збереження інформації. Telegram є лише зовнішнім
інтерфейсом. Бізнес-правила не залежать від `python-telegram-bot`.

## Цільова структура першого етапу

```text
src/personal_bot/
├── __main__.py                 # Точка входу
├── bootstrap.py                # Збирання та явне підключення модулів
├── config.py                   # Завантаження налаштувань із .env
├── core/                       # Спільні доменні моделі, enum-и, помилки
├── db/
│   ├── database.py             # Єдиний клас, що працює з sqlite3
│   ├── schema.py               # Початкове створення таблиць
│   └── repositories/           # По одному репозиторію на сутність
├── access/                     # Майбутні авторизація й заявки
├── categories/                 # Майбутня бізнес-логіка категорій
├── settings/                   # Майбутня логіка налаштувань
└── telegram/
    └── menus/                  # По одному файлу на меню
```

Реалізовані тільки модулі, потрібні для SQLite та мінімальної команди `/start`.
Категорії, налаштування й меню адміністратора з'являтимуться разом із першим
кодом, який їх використовує: код "про запас" не додається.

## Взаємодія шарів

```text
Telegram handler -> Service -> Repository -> Database -> SQLite
```

- Handler приймає Telegram-подію, передає прості дані в сервіс і показує
  результат користувачу.
- Service містить бізнес-правила та не імпортує Telegram-типи.
- Repository виконує запити лише через `Database`.
- Лише `Database` імпортує та використовує `sqlite3`.
- `Database.transaction()` атомарно фіксує зміни або повністю їх скасовує.

## Початкова структура бази даних

### `users`

Авторизовані користувачі: Telegram ID, дані профілю, номер, роль, статус і
дати створення/оновлення. `telegram_id` є унікальним ідентифікатором.

Ролі: `super_admin`, `admin`, `user`.

Статуси створеного користувача: `active`, `blocked`. Стан `pending` належить
заявці, бо користувач створюється тільки після схвалення.

### `access_requests`

Незмінна історія заявок. Вона містить знімок Telegram-даних, номер телефону,
стан (`pending`, `approved`, `rejected`) і дані про розгляд. Частковий
унікальний індекс гарантує лише одну `pending`-заявку для одного `telegram_id`.

### `user_settings`

Один запис на користувача: часовий пояс, валюта, формат дати, мова та дата
оновлення.

### `categories`

Особисте дерево категорій: власник, необов'язкова батьківська категорія, назва,
іконка та дати. Категорії не є бізнес-сутностями, зокрема автомобілями.

У сервісі категорій буде обов'язкова перевірка, що батьківська категорія має
того самого власника. SQLite-зовнішній ключ не може виразити цю складену
вимогу без надмірного ускладнення схеми.

## Авторизація: поточна поведінка

1. Telegram ID — головний ідентифікатор.
2. Команда `/start` шукає користувача за Telegram ID.
3. Невідомому користувачу показується кнопка надсилання власного номера.
4. Лише contact із `contact.user_id`, що збігається з Telegram ID відправника,
   приймається як підтвердження номера.
5. Перший користувач після створення порожньої БД стає `super_admin`, а його
   `users` і `user_settings` створюються в одній транзакції.
6. Для наступного користувача створюється одна pending-заявка; повторне
   надсилання контакту не створює дубліката.
7. Користувачеві зі статусом `active` показується головне меню.
8. Користувачеві зі статусом `blocked` показується повідомлення про блокування.

Розгляд заявок Super Admin ще не реалізований.

## Ізоляція даних

Кожен запит до категорій та майбутніх особистих сутностей обов'язково фільтрує
дані за `owner_user_id`. Це унеможливлює доступ до чужих записів через callback
або навмисно змінені параметри. Запити до `users` і `access_requests` є
винятком: вони працюють за Telegram ID або правами Super Admin, бо не мають
поля власника.

## Додавання нового модуля

Новий модуль створюється в окремій папці, наприклад `vehicles/`. У ній будуть
його сервіс, Telegram handlers і, за потреби, папка `repositories/`. Меню
створюється окремим файлом у `telegram/menus/`. Єдина зміна наявного коду —
явне підключення нового модуля у `bootstrap.py`.

Автоматичне виявлення модулів, plugin system і feature loader не
використовуються.

## Правила розробки

- Класи, методи, функції, змінні та файли називаються англійською без
  неочевидних скорочень.
- Повідомлення користувачу пишуться українською.
- Один файл має одну відповідальність.
- Не додавати абстракції, код або залежності, які не потрібні зараз.
- Не дублювати логіку; повторюваний код потрібно винести після окремого
  погодження, якщо це впливає на архітектуру.
- Нові залежності потребують попереднього пояснення й підтвердження.
- Нова схема БД у майбутньому вимагатиме окремого погодженого рішення щодо
  оновлення вже наявних баз даних.
